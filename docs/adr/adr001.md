# ADR : Utilisation d'une base de données distribuée

## Statut
**Accepté** - Décision prise dans le cadre de la migration vers une architecture entièrement distribuée dans le nuage.

## Contexte
L'application Store Manager a évolué d'un monolithe vers une architecture microservices event-driven. Actuellement, chaque service possède sa propre base de données et la synchronisation entre ces bases est gérée manuellement. 

Avec la migration prévue vers le nuage, nous devons préparer l'application pour un fonctionnement entièrement distribué. Les défis identifiés incluent :
- Gestion manuelle complexe de la synchronisation entre multiples bases de données
- Risques de cohérence des données entre les services
- Difficulté à garantir la haute disponibilité et la résilience
- Scalabilité limitée avec l'approche actuelle

Nous avons besoin d'une solution de base de données qui assure nativement la distribution et la synchronisation des données à travers un cluster distribué.

## Décision

Lors du labo 9, nous avons expérimenté avec deux bases de données distribuées : YugabyteDB et CockroachDB. Elles ont été choisies car ce sont des bases modernes, bien documentées, qui nécessitent moins de [CPU](https://docs.yugabyte.com/stable/deploy/checklist/#cpu-and-ram) et de [RAM](https://www.cockroachlabs.com/docs/stable/recommended-production-settings#node-density-testing-configuration) pour une configuration de base que d'autres solutions distribuées populaires, comme [Apache Cassandra](https://cassandra.apache.org/doc/latest/cassandra/managing/operating/hardware.html#memory).

Nous adoptons donc ____________ comme base de données distribuée pour l'application Store Manager.

**Justification :**
- **Distribution native** : la base de données gère automatiquement la réplication et la distribution des données à travers le cluster, éliminant la nécessité de synchronisation manuelle
- **Haute disponibilité** : Architecture avec réplication configurable garantissant la continuité de service
- **Scalabilité horizontale** : Permet d'ajouter facilement des nœuds pour augmenter la capacité sans interruption de service
- **Préparation cloud** : Architecture distribuée alignée avec les exigences d'un déploiement nuage multi-régions
- **Simplicité opérationnelle (dans un contexte distribué)** : Réduit la complexité de gestion en centralisant la logique de distribution au niveau de la base de données plutôt qu'au niveau applicatif
